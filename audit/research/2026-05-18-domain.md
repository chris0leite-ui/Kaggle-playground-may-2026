## Papers / writeups reviewed

- **Optimum Racing: A F1 Strategy Predictor using Reinforcement Learning** —
  IJRASET, Apr 2025. Deep Recurrent Q-Network (DRQN) trained on a Monte-Carlo
  race sim. State: tyre deg, gaps to competitors, race progress, weather,
  safety-car incidents. Action: compound choice + pit lap. Tested on unseen
  2021-mid2024 races. (S1)
- **A State-Space Approach to Modeling Tire Degradation in F1** — arXiv
  2512.00640 (2025). Bayesian SSM with **latent tyre pace** as hidden state;
  **fuel mass is an observable covariate** so deg is identified separately.
  Compound-specific deg rates not statistically distinct in their case study;
  skewed-t obs model fits better. Built on FastF1 lap timing. (S2)
- **Explainable Time Series Prediction of Tyre Energy in F1 Race Strategy** —
  ACM SAC 2025 / arXiv 2501.04067. Mercedes telemetry; XGBoost vs DL for
  forecasting **tyre energy** (a thermal proxy), with SHAP + counterfactuals. (S3)
- **TUM FTM race-simulation** — GitHub TUMFTM/race-simulation. Quasi-steady
  race sim with INI parameter files for ~121 F1 races 2014-2019. Models
  fuel-mass decrease, tyre deg, inter-driver interactions, MC sampling of
  random events (VSC/SC). Four Virtual Strategy Engineer variants
  (base / real / supervised / RL); dry compounds only. (S4)
- **Virtual Strategy Engineer (Heilmeier et al., Appl. Sci. 2020)** —
  MDPI 2076-3417/10/21/7805. Same TUM lineage; ANN strategy decisions; explicit
  list of input features the engineers think matter. (S5)
- **Pit-to-Win, UC Berkeley iSchool 2025** — practitioner-style writeup. (S6)

## Decision drivers from F1 strategy literature

- **Undercut gain** ~1.5-2.0 s on out-lap from fresh-tyre delta vs warming
  tyre; effective only if track allows fast warm-up. (S7,S8)
- **Pit-loss time per track**: ~20-25 s green-flag; e.g. Monaco ~20 s, COTA
  needs 18 s gap to hold, Silverstone needs ~27 s. (S9)
- **VSC pit cost** ~9-12 s (vs ~24 s green); SC similar or larger savings.
  Massive strategy lever. (S10)
- **Fuel-burn correction**: 0.025-0.040 s/lap per kg, ~0.3-0.4 s/lap per 10 kg;
  track-dependent (heavy-braking tracks hit harder). (S11)
- **FIA dry-race rule**: ≥2 different slick compounds → forces ≥1 pit. Monaco
  2025 is exception (≥3 sets). (S12)
- **Tyre deg curves**: compound-specific (Pirelli C1-C5 mapped to H/M/S per
  event); ~0.05-0.06 s/lap on Hamilton hard/medium in 2024 SSM fit (S2).
- **Track-evolution / rubbering-in** lowers lap times monotonically across
  the race, conflated with fuel burn unless modelled jointly.

## External data sources and join keys

- **FastF1** (docs.fastf1.dev). Laps DataFrame has Driver, LapNumber,
  Compound, TyreLife, Stint, LapTime, sector times, TrackStatus, IsPitOutLap.
  Pit timing via pit_in_time/pit_out_time. Join key for s6e5: would need
  `(Race name → Event, Driver code, LapNumber)`. (S13)
- **OpenF1** (openf1.org/docs). Endpoints + columns:
  - `laps`: driver_number, session_key, lap_number, duration_sector_1/2/3,
    lap_duration, i1/i2/st_speed, is_pit_out_lap.
  - `pit`: stop_duration, lane_duration, lap_number.
  - `stints`: stint_number, compound, lap_start, lap_end, tyre_age_at_start.
  - `weather`: air_temp, track_temp, humidity, rainfall, wind_speed/dir.
  - `race_control`: flag, category, message, lap_number (VSC/SC text in msg).
  - `intervals`: gap_to_leader, interval.
  - `position`: position per timestamp.
  Joins: (session_key, driver_number, lap_number). 2023+ free. (S14)
- **Ergast** (history, deprecated end-2024 but mirrored). Race-level pit-stop
  laps and durations 1996-2024.
- **Aadigupta Kaggle F1 Strategy Dataset** — lap-level multi-race; suggested
  external for this competition. Schema not exposed via API (page is JS-rendered)
  but advertised columns match FastF1: race, driver, lap, compound, tyre_life,
  pit_target. (S15)

**Match rate to s6e5 synth**: Driver (887 levels) far exceeds real F1 entry
pool (~30/year × 6 yrs ≈ 50 unique). Likely **Driver is a synthetic ID**, not a
real driver name → external joins by (Race, Driver) will not work directly.
Race (26 levels) ≈ one season. Match likely possible only at the (Race,
LapNumber)-aggregated level (per-track features like pit-loss, deg curve,
VSC base-rate). **This is fold-safe** because it's not target-derived.

## Features likely MISSING from our 14-column synth

- **Track pit-loss seconds** (per-race scalar). Why: directly sets the
  undercut/overcut break-even. Derive from FastF1: median(pit-out lap +
  pit-in lap) − median green lap. Per-Race lookup. (S9)
- **VSC / SC indicator on this lap** (TrackStatus / race_control). Drives a
  large fraction of strategic pits. The synth's "TrackStatus" may already be
  there — confirm; if absent we have a gap. Frontiers paper uses it. (S16)
- **Rainfall / wet-flag**. Forces pit irrespective of tyre age. Weather
  endpoint in OpenF1. Some races in train set may be wet; check distribution
  of unusually low TyreLife pits. (S14)
- **Gap to driver ahead / behind** (intervals). Undercut/overcut decisions
  are gap-driven. Approximate from Position ranks + LapTime cumsum within
  (Race, LapNumber). (S14)
- **DriverAheadPitted / DriverBehindPitted prev lap** (Frontiers feature).
  Constructable inside fold from existing (Race, LapNumber, Driver) panel
  without leakage if we shift by lap. (S16)
- **Compound life relative to expected life** (TyreLife / compound-typical-life).
  Lookup compound-typical life per race from Pirelli/FastF1 stints.
- **Stint number within mandatory-2-compound constraint**: "have I used my
  second compound yet?" If only one compound used and laps remaining < N → forced pit.
- **Fuel-corrected LapTime trend in stint** = LapTime + α·(LapsRemaining)
  with α≈0.03. Cleaner deg signal than raw LapTime. (S2,S11)
- **Race-progress × compound interaction**: compound expected life as
  fraction of race length, not absolute laps.
- **Pit-window distance** = abs(LapsRemaining − typical-second-pit-lap-for-this-race).
- **Qualifying / starting position** — proxy for car pace; absent here and
  not derivable. (S5)

## Heuristic rules to reverse-engineer (synth-generator hypotheses)

- **H1: threshold-on-(TyreLife / CompoundMaxLife)**. CTGAN may have learned a
  rule like "Soft and TyreLife>18 → high pit prob." Test: fit a single
  decision-stump per Compound on TyreLife; look for a sharp threshold;
  compare AUC contribution vs continuous monotone fit. <10 min CPU.
- **H2: race-progress S-curve**. Pit probability rises monotonically with
  RaceProgress until ~0.7 then falls (no pit near end). Test: isotonic fit
  of P(y=1 | RaceProgress) bucketed by compound. <5 min.
- **H3: "competitor pit cascade"**. Within (Race, LapNumber), if multiple
  drivers pitted last lap, current driver more likely to pit (undercut
  defence). Test: compute Race-Lap pit-rate at L-1; correlate to y; add as
  fold-safe feature (Rule 24 — recompute per fold). ~15 min.
- **H4: Stint-length quantile relative to race quantile**. Test: for each
  (Race, Compound) compute median stint length distribution; deviation
  → pit prob. ~15 min.
- **H5: VSC/SC TrackStatus override**. If TrackStatus indicates VSC/SC and
  TyreLife > min_threshold → pit prob spikes. Test: stratify by TrackStatus
  numeric codes; check class-rate jumps. ~5 min.
- **H6: Cumulative_Degradation threshold**. The synth already has this column.
  Test if it's monotone-step (CTGAN often discretises) → a single split-point
  recovers most signal. ~5 min.
- **H7: PitNextLap decoupling 19%** cases — likely lap-level noise injection
  by CTGAN. Calibrate per-row Bayesian shrinkage of model output toward
  per-(Race,LapNumber) base-rate. ~30 min stack-meta probe.
