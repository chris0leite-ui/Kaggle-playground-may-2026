## The actual goal of this whole project

Kaggle competitions are **benchmarks**, not the point.

The real point is: **learn how to do research, and semi-automated research,
with coding agents**.

- I'm the strategist / orchestrator of the orchestration.
- The coding agent (Claude / Claude Code) does the orchestration itself.
- I observe what works and what doesn't.
- Kaggle is convenient because it gives me an objective benchmark
  (leaderboard) to compare my work against.

It is **not** about building production ML. The competition setting is
laboratory-clean: stable data pipelines aren't required.

## Current friction (the thing that motivates writing this down)

A lot of friction in the operational setup. I'm trying to solve it by:
- working on **several branches** in parallel,
- across **several machines** via Claude cloud services (Claude Code
  running in containers),
- where the agents communicate **only through GitHub**.

The agents:
- pull data and submit through the **Kaggle API**,
- maybe call **TabPFN** (Prior-Fitted Networks) APIs,
- download Python packages as needed.

Concretely the friction shows up in:
- **communication** between parallel agents,
- **finding common ground** for taking decisions when they're working in
  parallel,
- coordinating without me having to manually transcribe state between
  them.

I myself **don't write code**. I only instruct and check.
